# YC Knowledge Base: The Technical Founder Playbook
*Source: Diana Hu - "How to Build and Succeed as a Technical Founder" (Startup School)*

## 1. The Role of the Technical Founder
- **Partner, Not Just a Dev:** A technical founder is a full partner in the startup journey, requiring intense commitment.
- **Lead Developer in Early Stages:** Responsible for putting the project together, building, and getting it out.
- **Full-Stack & Beyond:** Must do "all the tech things" (front-end, back-end, DevOps, UX, IT, even hardware/mechanical if applicable).
- **User-Centric:** Leads building *and* talks to users to get insights.
- **Bias for Action:** Prioritizes "good enough" over "perfect architecture." Comfortable with technical debt, inefficient processes, ugly code, and chaos.
- **"Whatever It Takes":** Committed to the success of the company, doing anything necessary.
- **Evolving Role:** Post-PMF, the role shifts from coding to scaling tech, hiring, and management (architect or VP Eng). Coding time decreases significantly.

## 2. Building Stages & Principles

### Stage 1: Ideating (Goal: Prototype)
- **Goal:** Build a prototype as soon as possible (days) to show and demo to users.
- **Focus:** Singular focus on demo; it doesn't need to work fully.
- **Method:** Use prototyping software (Figma, Invision), simple scripts, 3D renderings (for hard tech).
- **Mistakes to Avoid:**
  - **Overbuilding:** Don't aim for a full MVP at this stage.
  - **Not Talking to Users:** Show the duct-taped prototype and get feedback.
  - **Attachment to Ideas:** Be willing to let go of bad ideas based on user feedback.

### Stage 2: Building an MVP (Goal: Launch)
- **Goal:** Build to launch quickly (weeks, sometimes months for hard tech). Get user commitment (ideally payment).
- **Mistakes to Avoid:**
  - **Hiring Too Early:** Slows down launch, founders miss key learnings and insights from building.
- **Principles:**
  - **"Do Things That Don't Scale" (Paul Graham):** Find clever hacks. Manually onboard users, provide insane custom support. (e.g., Stripe founders manually processing payments).
  - **"90/10 Solution" (Paul Buchheit):** The first version is not the final. Push off features to post-launch. Restrict product to limited dimensions (geo, data type, functionality, user type).
    - *Example:* DoorDash (Palo Alto Delivery) started with PDFs for menus, Google Forms/Docs for backend, Find My Friends for drivers. Focused on Palo Alto.
  - **Choose Tech for Iteration Speed:** Use what you're dangerous with and comfortable launching quickly. Leverage third-party frameworks/APIs (Auth0, Stripe, AWS, Firebase, Webflow). Avoid building from scratch.
    - *"Jedi Master" Engineers:* Choose simple, fast tech (PHP, JavaScript, Python/Django) over "optimal" big-company tech (Kafka, Kubernetes).
    - *Example:* Facebook scaled PHP with HipHop. WayUp chose Django/Python over Ruby/Rails for speed.
  - **Tech Choices Don't Matter as Much (Early):** If you get users, you can solve tech debt later. The only tech choices that matter are tied to customer promises (e.g., API level).

### Stage 3: Launch (Goal: Iterate to Product-Market Fit)
- **Goal:** Continuously iterate towards PMF.
- **Principles:**
  - **Quickly Iterate with Data:** Use hard data (analytics, KPIs) and soft data (user interviews) to understand why users stay or churn.
  - **Continuously Launch:** Keep adding features and iterating. (e.g., Segment's multiple launches).
  - **Balance Building vs. Fixing:** Tech debt is fine. Don't over-fix bugs or refactor too early. "Feel the heat of that tech." (e.g., Pokémon Go launched with broken login but still succeeded).
- **Mistakes to Avoid:**
  - **"What Would Google Do?":** Don't build like a big company or hire too quickly.
  - **Focusing Only on Fixing/Refactoring:** Prioritize building new features that drive PMF.
  - **Not Talking to Users:** Technical founders must remain involved in understanding user insights.
  - **Building Features Without Growth:** Pair engineering with sales/growth to find growth hacks.

### Stage 4: Post-Product-Market Fit (Role Evolution)
- **Focus:** Scale tech, address technical debt, make strategic hires.
- **Hiring:** More focused hiring, often starting with people you know.
- **Role Shift:** Less coding (e.g., 70% coding at 2-5 engineers, <50% at 5-10, almost none beyond 10). Role morphs into architect or VP Eng.

## 3. Key Mantras
- "Startups move quickly."
- "You gotta do it." (No "not in my pay grade" mentality).
- "Don't try to be the kind of like cool engineer, just build things from scratch. No, just use all these frameworks."
